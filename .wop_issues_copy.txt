===== ISSUE #160: Voice recording silently fails when browser mic permission isn't granted (Android, DuckDuckGo) =====
LABELS: bug
CREATED: 2026-05-28T16:53:25Z

## Symptom (reported)

User on Android using the DuckDuckGo browser:

1. Started a voice entry recording. DuckDuckGo asked for microphone permission; user chose **"Ask every time"**.
2. On the next voice-entry attempt, the browser **did not re-prompt** for permission — but recording also silently failed (nothing was captured).
3. The user only diagnosed it by going into the Android app settings for DuckDuckGo and switching the OS-level mic permission from "Ask every time" to "While using the app".

## Problems

**Bug:** When the browser denies / fails to grant mic access at recording start, Loore doesn't surface any error. The user sees the UI behave normally but no audio is captured — same data-loss class as #88 / #127 (silent drop), just with a different root cause.

**Friction:** User comment — "It's annoying to have to grant whole-browser mic permission just to occasionally record a voice memo." That's a platform-level constraint Loore can't directly remove, but we can at least handle it gracefully and tell the user what's happening.

## Proposed fix

Two parts:

### 1. Detect mic permission at recording start

Before / during `MediaRecorder` start, check `navigator.permissions.query({ name: 'microphone' })` where supported. If the state is `denied` or the `getUserMedia` call fails with `NotAllowedError` / `PermissionDeniedError`:

- Stop the recording attempt immediately (don't enter the "recording" visual state).
- Show a clear, actionable error toast/modal explaining:
  - Mic permission was not granted.
  - On Android, the user may need to update the browser's app-level mic permission in OS settings (link or short instructions).
  - On iOS / desktop, the user can re-request via the browser's site-permissions UI.

### 2. Distinguish "permission denied" from "no audio captured" in the recovery path

The voice-recovery banner currently helps when a recording was interrupted, but it can't help when nothing was ever recorded. Make sure the permission-denied path leaves no orphaned draft / stuck "recording…" UI.

## Reproduction

Android + DuckDuckGo browser (likely reproducible on other mobile browsers that default to "ask every time" — Firefox, Brave, Samsung Internet):

1. Revoke mic permission in browser app settings.
2. Try to start a voice entry in Loore.
3. Expected: clear error explaining the situation.
4. Actual: silent failure — UI looks fine, no audio captured.

===== ISSUE #141: Voice transcription drops chunks on highly repetitive content (e.g. singing) =====
LABELS: bug
CREATED: 2026-05-28T16:40:07Z

## Symptom

`gpt-4o-transcribe` drops chunks of audio when the input is highly repetitive — singing, repeated phrases, mantras, etc. This is a known failure mode of large transcription models (they collapse into "I've already transcribed this" mode and emit nothing).

- Surface a small UI indicator if a chunk used the fallback path, so the user knows transcription quality may be slightly lower there.

===== ISSUE #140: Sentence-aware TTS chunking — fall back to word-aware for tiny first chunks =====
LABELS: enhancement

## Problem

===== ISSUE #137: Claude import: handle oversized zip files (port the ChatGPT-import fix) =====
LABELS: bug
CREATED: 2026-05-28T16:39:47Z

## Problem

The Claude archive import currently uploads the full zip file to the backend, which fails (or risks OOM) on large archives — same problem the ChatGPT import had before commit `f6d329e`.

`frontend/src/components/ImportData.js:179` (`handleClaudeImportFile`) — POSTs the raw zip to `/import/claude/analyze` as `multipart/form-data`. ## Solution: port the ChatGPT fix

2. **Backend (`import_data.py`)** — change the `/import/claude/analyze` endpoint to read a JSON file field directly (no zip handling). The ChatGPT path already does this; mirror it for Claude.

===== ISSUE #130: LLM has no temporal grounding — confabulates relative times (e.g. "2 days ago" for same-morning event) =====
LABELS: bug
CREATED: 2026-05-28T16:27:48Z

## Symptom

User report:

> Loore doesn't seem to be able to track time. Like I talked to it about something this morning, then later I started a new conversation and it referenced that thing \"you were just talking about X 2 days ago\", and I've seen similar time wonkiness before.

## Root cause

The LLM is given **no per-message timestamp information** and **no current-date anchor** in any prompt. From the model's perspective every message appears to be happening \"now\" with no chronological ordering.

**Audit findings:**

- `backend/tasks/llm_completion.py:922` — each node going into chat context is rendered as just `\"author <username>: <content>\"`. `created_at` is never attached.
- `backend/prompts/agentic.txt` and other system prompts — no \"today is …\" / current-time injection anywhere.
- The only place time appears is in the `{user_recent_raw}` block (`backend/utils/tokens.py:48–62`) as coarse range headers like `[Covers 2026-03-10 to 2026-03-28]` wrapping bulk text — not per-message annotations.
- Where dates do appear, they're naive UTC (see #128). A message written at 11pm CEST shows up as the next day's date, which alone could plausibly drift the model's \"how long ago\" sense across midnight UTC.

With no temporal grounding, the model is left to reconstruct \"when\" from content alone and confabulates phrases like \"2 days ago.\"

## Fix

Annotate **every message** sent to the LLM (user messages, assistant messages, system messages, retrieved context — all of them) with a uniform absolute-time prefix in the **user's local timezone**.

### Design constraints (from the bug-report discussion)

- **Don't single out the system prompt.** No \"Today is X\" anchor injected separately — treat every message the same way. The model derives \"now\" from the most recent message's timestamp like any other temporal information.
- **User's local timezone, not UTC.** UTC stamps require the model to know the user's offset, which it doesn't.
- **Absolute times only — no relative phrasing** like \"yesterday\" / \"3 days ago\". Reason: a stored message that was \"3 days ago\" at write-time will be re-served later as context for a new conversation, and the relative phrase will then be wrong/misleading. Absolute timestamps stay correct forever.

### Suggested format

Each message prefixed with an ISO-ish local stamp, e.g.:

```
[2026-05-28 09:14 CEST] author hrosspet: I'm going to focus on the timestamps bug today.
[2026-05-28 14:32 CEST] assistant: Sounds good — want me to start with …
```

Day-of-week could be added if it helps disambiguate (`[2026-05-28 Thu 09:14 CEST]`) — that's still absolute, just more readable.

## Open sub-question: where does the user's timezone come from?

There is **no `User.timezone` field** today (`grep` of `backend/models.py` and routes returns nothing). Two options:

1. **Add `User.timezone` column.** Capture from `Intl.DateTimeFormat().resolvedOptions().timeZone` in the frontend on signup / first session and on user-initiated change. Standard, persists, lets the backend format independently of any single request.
2. **Pass timezone on every LLM-triggering request.** Frontend includes `Intl.DateTimeFormat().resolvedOptions().timeZone` as a header or body field on each chat/voice submission. No schema change, but every entry point needs to thread it through.

Option 1 is cleaner long-term. Either works for fixing the bug.

## Related

- #128 — UTC timestamp display in UI. Same underlying \"we don't track user-local time anywhere\" gap. Both issues should likely be solved by the same `User.timezone` plumbing.

## Implementation notes

Worth doing as a tiny shared hook so the behavior stays consistent:

```js
// frontend/src/hooks/useSubmitShortcut.js
export function useSubmitShortcut(ref, onSubmit, enabled = true) {
  useEffect(() => {
    const el = ref.current;
    if (!el || !enabled) return;
    const handler = (e) => {
      if (e.key === 'Enter' && (e.metaKey || e.ctrlKey)) {
        e.preventDefault();
        onSubmit();
      }
    };
    el.addEventListener('keydown', handler);
    return () => el.removeEventListener('keydown', handler);
  }, [ref, onSubmit, enabled]);
}
```

Each surface attaches it to its textarea ref + calls the same handler the submit button calls. Disable when the submit button would be disabled (e.g. empty content, in-flight request).

===== ISSUE #128: Timestamps display in UTC instead of user's local timezone (backend emits naive ISO strings) =====
LABELS: bug
CREATED: 2026-05-28T16:14:27Z

## Problem

User-facing timestamps across the app are effectively shown in UTC even though most components call `toLocaleString()` / `toLocaleDateString()`. A user in CEST (UTC+2) sees a node created at 14:32 UTC rendered as `5/28/2026, 2:32 PM` instead of `5/28/2026, 4:32 PM`.

## Root cause

The bug is on the **backend serialization side**, not the frontend.

- `backend/models.py` stores timestamps via `datetime.utcnow()` (naive datetimes, no `tzinfo`) into `db.DateTime` columns that don't have `timezone=True`.
- All serializers call `.isoformat()` on those naive datetimes, producing strings like `"2026-05-28T14:32:00.123456"` — **no `Z` suffix and no UTC offset**.
- Per ECMA-262, when JavaScript's `new Date(...)` parses an ISO string *without* a timezone indicator, it interprets it as **local time**, not UTC.
- The frontend then formats that (incorrectly parsed) `Date` with `toLocaleString()`. Result: UTC values displayed as if they were already local time → systematically off by the user's timezone offset.

Example: a CEST user sees the raw UTC value, which looks identical to a "UTC timestamp leaking into the UI."

## Affected surfaces

Pretty much everywhere a timestamp is shown. Quick audit:

- `frontend/src/components/NodeFooter.js:18` — node creation time (Feed, node detail, lists)
- `frontend/src/components/Dashboard.js:349` — \"Generated on\" line on profile
- `frontend/src/components/VersionHistoryDrawer.js:36` — version history dates
- `frontend/src/components/SearchModal.js:127` — search result dates
- `frontend/src/pages/AiPreferencesPage.js:132,159` — AI preferences versions
- `frontend/src/pages/ProfilePage.js:193` — profile version dates
- `frontend/src/pages/TodoPage.js:338,365` — todo/checklist versions
- `frontend/src/pages/PromptDetailPage.js:206,232` — prompt versions
- `frontend/src/pages/PromptsPage.js:114` — prompts list

(Backend export filenames like `write-or-perish-export-<ISO>.txt` also encode UTC, but that's intentional and not user-facing in the same way.)

## Proposed fix

Pick one of these — backend is cleaner because it fixes the bug at the source:

**Option A — Backend (preferred):** Emit ISO strings with explicit UTC marker so `new Date()` parses them as UTC.

- Cheapest patch: append `\"Z\"` (or use `.isoformat() + \"Z\"`) in serializers — touches `backend/utils/serialization.py`, `backend/routes/feed.py`, `backend/routes/ai_preferences.py`, `backend/routes/export_data.py`, `backend/utils/quotes.py`, `backend/routes/terms.py`, `backend/routes/import_data.py`, `backend/tasks/recent_context.py`, and any other `.isoformat()` call sites on timestamp columns.
- Cleaner long-term: switch `datetime.utcnow()` → `datetime.now(timezone.utc)` and (optionally) migrate `db.DateTime` columns to `timezone=True`. Then `.isoformat()` naturally produces `\"2026-05-28T14:32:00+00:00\"`.

**Option B — Frontend workaround:** Centralize date formatting in a single util (`frontend/src/utils/date.js`) that appends `\"Z\"` before parsing if the string has no timezone marker. There's currently **no shared date util** — ~7 components each define their own `formatDate()` helper, so consolidating is worth doing regardless of the timezone fix.

Recommend doing **both**: Option A for correctness, plus the consolidation from Option B to kill the duplicated `formatDate()` helpers and standardize the relative-time UX (\"today\"/\"yesterday\" is only implemented in one of the seven copies).

## Verification

After the fix, set the browser to a non-UTC timezone (e.g. via DevTools → Sensors → Location → Timezone override), create a node, and confirm the displayed creation time matches the wall clock in that timezone.

===== ISSUE #124: Recovery flow: resumed recordings drop pre-resume audio at finalize merge =====
LABELS: 
CREATED: 2026-05-06T22:03:41Z

## Symptoms

When a recording is interrupted (page refresh) and the user clicks "Continue" on the recovery banner, the resumed session correctly continues incrementing the timer, but the final transcribed message contains **only the post-resume audio**. Pre-resume audio is lost from the transcript despite those chunks being safely uploaded to the server.

## Reproduction

On staging or local:

1. Start a voice recording (any URL — both `?force_mime=mp4` and `?force_mime=webm` reproduce, as does the default WebM negotiation).
2. Record for ~40s so at least one timeslice chunk uploads (chunks 0 and 1 land at 15s and 30s).
3. Refresh the page mid-recording.
4. On the recovery banner, click "Continue recording". Timer correctly resumes from ~30s.
5. Record additional audio.
6. Stop.

**Expected:** final node's transcript contains both pre- and post-resume audio.

**Actual:** transcript contains only the post-resume portion. Pre-resume chunks (0 and 1 in the example) are stored on disk but their content is missing from the final draft/node.

Confirmed on staging for **both** `?force_mime=mp4` and `?force_mime=webm` on macOS Safari (#123 testing). Pre-existing — not introduced by #123, which only made the bug more visible by exposing the MP4 codepath.

## Root cause

When the user clicks "Continue", the frontend constructs a **fresh `MediaRecorder` instance** with a new mic stream (`useStreamingTranscription.resumeStreaming` → `useStreamingMediaRecorder.startRecording`). That recorder produces its own complete media stream from scratch:

- For fMP4: the first emission from the resumed recorder starts with its own `ftyp` + `moov`, followed by `moof` + `mdat` fragments.
- For WebM: the first emission starts with its own EBML header + Segment + Tracks, followed by Cluster fragments.

The original chunks 0..N-1 belong to a **different** MediaRecorder instance with a different `moov` / different track UIDs / different timestamp epoch. They're not byte-concat-compatible with the resumed chunks.

At finalize time, `concat_fragmented_media` binary-appends the cached init segment + all chunks in order (`backend/utils/webm_utils.py:concat_fragmented_media`, `backend/tasks/streaming_transcription.py:transcribe_chunk_batch`). For a resumed session this produces:

```
[init.{ext}] + [chunk_0 (orig stream)] + [chunk_1 (orig stream)] +
[chunk_2 (RESUMED stream w/ own ftyp+moov / EBML)] + [chunk_3 (resumed stream)] + ...
```

The merged file has two unrelated init segments mid-stream. ffmpeg's remuxer reads the second one and outputs only what it describes, so the pre-resume audio is silently dropped during the `-c copy` remux.

WebM appears more tolerant in some cases because Matroska's demuxer permits concatenated Segments, but the underlying architectural problem is identical for both formats — confirmed empirically.

## Proposed fix

Treat each `MediaRecorder` instance as a "subsession" and transcribe subsessions independently, concatenating their transcripts into `draft.content`.

**Backend changes (~50 lines):**

1. **Detect init-bearing chunks at upload** (`backend/routes/drafts.py:upload_streaming_chunk`). For chunks N>0, peek at the first 8 bytes:
   - MP4: `ftyp` at offset 4 → init-bearing (subsession boundary)
   - WebM: `0x1A45DFA3` at offset 0 → init-bearing
   
   If init-bearing, call `persist_init_segment` with an index suffix so it writes `init.{N}.{ext}` (instead of overwriting `init.{ext}`). Chunk 0 keeps writing `init.{ext}` for backward compat.

2. **Track subsession boundaries** when batching (`transcribe_chunk_batch`). For a batch with chunk indices `[a, b, c, ...]`, find all `init.{X}.{ext}` files where `X` ∈ batch indices. Each X is a subsession boundary. Split the batch into sub-batches at those boundaries.

3. **Transcribe each sub-batch independently** with its own init prefix (`init.{X}.{ext}` or `init.{ext}` for the chunk-0 subsession). Concatenate transcripts into `draft.content` in chunk-index order — the existing per-chunk text storage already handles ordering.

**Frontend changes:** none. The recovery flow's resume signature is correct as-is — the backend handles the split.

## Scope

- Out of scope for #123, which addresses the iOS-Safari-no-WebM bug. That PR has been validated end-to-end (7-min Safari MP4 recording + 25s regression on multiple browsers) and should ship to unblock affected users.
- This recovery bug is pre-existing and affects all formats. Fix can be a focused follow-up PR with its own tests (subsession-boundary detection on real fMP4 + WebM resumed-chunk samples, batch partitioning correctness).

## Affected files (for the future fix)

- `backend/routes/drafts.py` — `upload_streaming_chunk` (init detection + suffixed persistence)
- `backend/utils/webm_utils.py` — `persist_init_segment` (accept optional index suffix)
- `backend/tasks/streaming_transcription.py` — `transcribe_chunk_batch` (sub-batch partitioning + per-subsession merge)
- `backend/utils/audio_storage.py` — broaden init filter to handle `init.{N}.{ext}` variants (already handles `init.*`)

===== ISSUE #78: Block SPA navigation during active recording (requires createBrowserRouter migration) =====
LABELS: 
CREATED: 2026-03-17T15:25:42Z

## Problem

When a user is actively recording on the Reflect or Orient page, clicking an in-app link (e.g. the Loore logo, nav items) navigates away without warning, losing the recording session.

The `beforeunload` event only fires for browser-level actions (refresh, close tab, external URL). React Router SPA navigation bypasses it entirely.

## What we tried

- `useBlocker` from react-router-dom v6 — throws because it requires `createBrowserRouter` (data router API)
- `unstable_usePrompt` — calls `useBlocker` internally, same error
- Both are incompatible with `<BrowserRouter>` which the app currently uses

## Current mitigation

- `beforeunload` dialog covers browser refresh/close
- Recovery banner on Reflect/Orient detects interrupted recordings and offers "Continue recording" / "Discard"
- At most 15s of audio (one chunk interval) is lost

## Suggested fix

Migrate from `<BrowserRouter>` to `createBrowserRouter` + `<RouterProvider>`. This unlocks the data router API including `useBlocker`.

### Scope

- Convert ~30 route definitions in `App.js` from JSX `<Route>` elements to a `createBrowserRouter([...])` config array
- Extract `<App>` wrapper logic (navbar, auth, modals) into a layout route component that renders `<Outlet />`
- Replace `<BrowserRouter><App /></BrowserRouter>` in `index.js` with `<RouterProvider router={router} />`
- `ProtectedRoute`, `Navigate` redirects, and `useNavigate` calls continue to work unchanged
- Add `useBlocker` to `useVoiceSession` hook to block SPA navigation during recording

### Effort

Moderate — mostly mechanical refactoring of route definitions. Core app logic stays the same.

===== ISSUE #32: Feature: Incremental user profile generation with context window management =====
LABELS: enhancement
CREATED: 2025-12-03T12:26:37Z

## Feature Request

### Summary
Improve user profile generation by making it incremental - using previously generated profiles as a base and updating them with new user data. This approach should be context-window-aware and handle users with data exceeding model limits.

### Motivation
Currently, profile generation likely processes all user data at once, which:
1. Doesn't scale for users with large amounts of content
2. Regenerates the entire profile from scratch each time
3. May hit context window limits for users with extensive writing

### Proposed Implementation

#### Core Logic
When generating a user profile:

1. **Check for existing profile**
   - If no profile exists → Initial generation mode
   - If profile exists → Incremental update mode

2. **Initial generation mode** (no existing profile)
   - Take first [N tokens ~ given model's context window limit] of user data (minus buffer for prompt/output - use current estimation logic)
   - Generate initial user profile
   - Store profile in DB with metadata (timestamp, tokens_processed, last_node_id, etc.)
   - If more data remains:
     - Take next [N token] chunk
     - Update profile based on new chunk + previous profile
     - Store updated profile
     - Repeat until all user data processed
   - Store all intermediate profiles in DB for history/debugging

3. **Incremental update mode** (profile exists)
   - Load most recent profile
   - Fetch only user data created since last profile generation (using timestamp/node_id)
   - If new data fits in context window:
     - Update profile with: previous profile + new data
   - If new data exceeds context window:
     - Process in chunks as in initial mode
   - Store updated profile with new metadata

#### Context Window Management
- Make aware of selected model's context window limit (e.g., Opus 4.5: 200k tokens)
- Use current token estimation logic with appropriate buffer (X tokens)
- Calculate: `usable_tokens = context_window - prompt_overhead - buffer - output_reserve`
- Chunk user data to fit within `usable_tokens`

#### Database Schema Considerations
Store profile metadata:
- `user_id`: which user
- `generated_at`: timestamp
- `model_used`: which LLM model
- `tokens_processed`: cumulative tokens used
- `last_node_id` or `last_timestamp`: track data boundary
- `profile_version`: incremental counter
- `is_latest`: boolean flag for current profile
- `profile_content`: the actual generated profile

### Benefits
1. **Scalability**: Handle users with 1M+ tokens of data
2. **Efficiency**: Only process new data after initial generation
3. **Workaround for limits**: Enable profile generation even when total data exceeds context window
4. **History**: Keep intermediate profiles for debugging/analysis
5. **Better profiles**: Iterative refinement should produce more coherent, comprehensive profiles

### Example Scenario
User with ~1M tokens of data, using Opus 4.5 (200k context):
1. First generation: Process 5 chunks of ~180k tokens each (with buffer)
2. Generate profile v1 from chunk 1
3. Update to profile v2 using v1 + chunk 2
4. Update to profile v3 using v2 + chunk 3
5. Continue until profile v5 (all data processed)
6. Two weeks later: User creates 20k tokens of new content
7. Next generation: Update profile v5 with just the new 20k tokens → profile v6

### Related Files
- Profile generation prompt/logic (location TBD)
- Database models for storing profiles and metadata
- Token estimation utilities
- User data fetching logic

### Technical Considerations
- How to order/prioritize user data for chunking (chronological? by node importance?)
- Prompt design for "update this profile with new information" vs "generate initial profile"
- Storage/retrieval of intermediate profiles
- UI to show users which data has been included in their current profile
